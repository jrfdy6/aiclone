# Fusion DC Admissions Playbook v1

## Status

This is a `recommended-default` playbook synthesized from:

- the Fusion OS operating model
- the Fusion outreach and admissions prompt library
- your corrections about sessions, package mix, pro relationships, and campus fit
- existing Fusion research in the workspace

It is designed to give your system usable decision logic now, without pretending we already know every hard threshold.

Where exact numeric thresholds are unknown, this playbook uses decision rules instead of fake precision.

## Core Principle

Fusion DC should not be interpreted like a generic admissions funnel.

The system should evaluate outreach and admissions through this lens:

1. Are we building enough future demand through the right pros?
2. Are the right families moving through the journey?
3. Are those families converting into the right package mix?
4. Are those packages producing enough full-time sessions to keep the campus healthy?
5. Are cost and campus social confidence blocking otherwise good-fit families?

## 1. Mission And Decision Context

### Recommended Default Answer

The data should most often inform these decisions:

1. where to reallocate outreach time
2. which pro relationships to deepen, protect, or deprioritize
3. whether a DAO needs coaching on targeting, follow-up, or conversion execution
4. whether a weak result is an execution problem, a structural constraint, or normal seasonality
5. how to set near-term campus goals without losing long-term pipeline health

When data conflicts, use this priority order:

1. `right-fit enrollments and full-time session health`
2. `future pro pipeline health`
3. `relationship equity with strategically important pros`
4. `activity volume`

If volume is up but conversion is down, do not assume failure.

Check in this order:

1. did quality of demand decline?
2. is the work still early-stage and building future yield?
3. are cost or social-fit objections rising?
4. is there weak follow-up after good meetings or tours?

### System Rule

The system should optimize for `full-time health plus future demand creation`, not just current-period volume.

## 2. Definitions And Success Criteria

### Recommended Default Answer

Do not redefine CRM-native stage names unless an admin confirms them.

Use existing system definitions for `Inquiry`, `AM`, `Register`, and `LPR`, but interpret them with Fusion logic.

Default success definitions:

- `Inquiry -> AM` success means qualified interest is moving forward without drag, backlog, or weak qualification.
- `AM -> Register` success means good-fit families are becoming paying, scheduled students, with special weight on full-time outcomes.
- `Outreach -> LPR` success means targeted professional relationship activity is producing real referral leverage, not just logged touches.

Default benchmark logic:

- `strong`: above recent trend, supported by downstream quality, with no hidden fragility
- `acceptable`: within seasonally normal range and supported by stable leading indicators
- `concerning`: below recent trend for more than one review cycle and confirmed by downstream weakness or quality decay

Metrics that should be visible but should not drive decisions alone:

- speed to contact
- raw outreach count
- raw registered student count
- student headcount without package mix

One metric the system should never read in isolation:

- `registered students`

Why:
It can hide weak full-time mix or too much tutoring / mentoring concentration.

### System Rule

The system should score metrics as `strong`, `acceptable`, or `concerning` using trend context, not standalone count logic.

## 3. Time Horizon And Pattern Sensitivity

### Recommended Default Answer

Use three default time windows:

1. `last 2 weeks` for tactical execution checks
2. `last 6 weeks` for trusted operating trend judgment
3. `quarter` for strategic direction and goal pacing

Use `YoY` only when the comparison is structurally fair.

YoY should be treated as weak or misleading when:

- staffing changed materially
- campus maturity changed materially
- capacity changed materially
- pricing or package strategy changed
- the comparison period has unusual seasonality

Known distortions to respect:

- summer is slower and more tolerant of part-time mix
- testing and late-spring timing can distort family decision speed
- mid-year crises can create urgent but irregular inquiry spikes
- relationship outreach may lag referral yield by weeks or quarters

### System Rule

Default to a `6-week trend lens` unless the user explicitly asks for a shorter or longer horizon.

## 4. Campus And Role Differences

### Recommended Default Answer

Campuses should not be ranked naively when they differ on:

- market maturity
- pro portfolio maturity
- staffing stability
- capacity
- local social proof
- local brand strength

Role differences that should change interpretation:

- `DAO / DO`: should be judged heavily on pro strategy, outreach quality, meetings, follow-up, and pipeline building
- `HOS`: should be judged more on campus credibility, culture, staffing readiness, and conversion support
- `new staff`: should be evaluated more on discipline and pattern quality than on mature referral output
- `seasoned staff`: should be expected to show stronger relationship leverage and cleaner conversion

Do not call a campus weak until you rule out:

- structural capacity constraint
- low social density / small-campus perception issue
- immature pro network
- staffing limitation

### System Rule

The system should separate `execution issues` from `structural constraints` before recommending intervention.

## 5. Outreach Philosophy

### Recommended Default Answer

High outreach volume is good only when it is targeted and relationship-building.

Quality outreach means:

- focused on the right professional segments
- producing meetings, responses, or event partnership
- followed by disciplined follow-up
- connected to long-term relationship value, not just same-week yield

Default segment priority for Fusion DC:

1. private school administrators
2. education consultants
3. therapists and psychiatrists
4. treatment center administrators
5. advocates and attorneys as strategic adjacency within consultant-like relationship work

Default segment classification:

- `priority`: private school administrators, education consultants, strongest therapists
- `maintenance`: existing active referrers, politically important relationships, stable but lower-volume partners
- `long-term pipeline`: cold strategic segments, treatment centers, underdeveloped therapist networks

Outreach behaviors most likely to produce LPRs:

1. meetings with the right pros
2. disciplined email follow-up after meetings
3. repeated relationship touches over time
4. event partnership and credibility-building
5. targeted outreach to pros already adjacent to strong-fit student profiles

### System Rule

The system should reward `meeting quality and follow-through`, not just activity count.

## 6. Referral And Professional Strategy

### Recommended Default Answer

Relationships to protect even when short-term volume dips:

- power-referring pros
- private school administrators with strong-fit history
- high-trust consultants who send aligned families
- any locally important relationship with reputational or network spillover

Strategic but still underdeveloped segments:

- therapist and psychiatrist networks that align to anxious or neurodivergent students
- treatment centers with likely long-lag value
- advocates / attorneys where family complexity requires trusted navigation

Default lag interpretation:

- same-week yield is too short for judging strategic outreach
- `2 weeks`: good for testing execution discipline
- `6 weeks`: good for testing whether outreach is creating momentum
- `1 quarter`: better for judging relationship-building yield

Politically important relationships should be protected even if immediate ROI is soft when they influence:

- local reputation
- counselor / school referral perception
- access to future-fit families

### System Rule

The system should rank relationships by `current output + future strategic value`, not by current output alone.

## 7. Risk Tolerance And Intervention Triggers

### Recommended Default Answer

Automatic trigger candidates:

- pro referrals decline across more than one review cycle
- full-time sessions registered flatten or decline while small-package mix rises
- meetings increase but referred inquiries or LPRs do not follow over a reasonable lag window
- cost objections or social-fit objections spike after tours
- too much dependence on one or two power referrers
- scheduled sessions weaken enough to threaten staffing logic

Default rule for acceptable decline:

- a one-period dip without corroborating weakness is usually `observe`
- a repeated decline plus downstream weakness is `intervene`
- a sharp decline in a critical leading indicator is `intervene now`

Default coaching-versus-system logic:

- `coaching`: weak targeting, weak follow-up, weak meeting discipline, weak tour-to-package execution
- `system change`: multiple people showing the same failure mode, bad stage definitions, dashboard blindness, or unrealistic goals
- `structural watch`: capacity, campus size perception, market maturity, local brand weakness

### System Rule

Do not escalate on a single weak number unless it is both material and a leading indicator with downstream risk.

## 8. Output Preferences

### Recommended Default Answer

Default answer style:

1. executive summary first
2. short bullets second
3. recommendations last

Default framing:

- `risk-first` for leadership review
- `opportunity-first` for planning and growth work

Default response contents:

- what matters
- what does not matter yet
- whether this is signal or noise
- likely cause
- one to three next actions

When requested, the system should also be able to return:

- coaching brief
- coaching script
- relationship-priority list
- goal-setting recommendation

### System Rule

The system should default to concise, decision-oriented analysis with actions included.

## Default Interpretation Logic

When asked, "What should I pay attention to?", the system should analyze in this order:

1. pro referral health
2. referral inquiry and LPR momentum
3. AM and Register conversion quality
4. full-time session health
5. cost and social-fit friction
6. structural constraints

When asked, "Is this a problem?", the system should answer in this order:

1. is it real signal?
2. is it seasonal?
3. is it structural?
4. is it execution?
5. does it cross intervention logic?

When asked, "What should I do?", the system should recommend from this order:

1. protect strong strategic relationships
2. reallocate outreach time
3. tighten follow-up behavior
4. address tour/package friction
5. adjust goals if the constraint is structural

## Recommended Solutions To Apply Inside Fusion OS

### 1. Add A Decision Layer Above Raw Metrics

Every admissions review should classify each pattern as:

- `intervene now`
- `monitor`
- `ignore as noise`

### 2. Add A Pro Segment Strategy Layer

Each pro or segment should be tagged:

- `priority`
- `maintenance`
- `long-term pipeline`
- `politically important`

### 3. Add A Metric Trust Layer

Each metric should be tagged:

- `primary decision metric`
- `supporting context metric`
- `vanity if isolated`

### 4. Add A Structural Constraint Check

Before the system criticizes a campus or operator, it should check:

- staffing
- capacity
- campus social density
- market maturity
- pro portfolio maturity

### 5. Add A Conversion Friction Layer

For AM -> Register analysis, the system should explicitly check:

- cost friction
- campus social confidence
- package fit
- scheduling lag

### 6. Add A Future Demand Layer

The system should always surface:

- referral inquiry trend
- pro meeting trend
- power-referrer concentration risk
- underdeveloped strategic segments

### 7. Add A Session Health Layer

The system should never summarize campus health without:

- full-time sessions registered
- small-package concentration risk
- scheduled session trend

## Recommended Default Prompt Preamble

Use this before running analysis against outreach and admissions data:

```md
Use Fusion DC Admissions Playbook v1.
Interpret the data as a session-based, pro-relationship-driven business, not a generic admissions funnel.
Prioritize full-time health, future demand creation, and strategically important professional relationships.
Separate signal from noise.
Separate execution issues from structural constraints.
Do not use raw activity or raw registration counts in isolation.
Only escalate issues that cross playbook intervention logic.
```
