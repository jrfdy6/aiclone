# AGC Capability Evidence Model

AGC must describe capability in a way that is credible now, not aspirational fiction.

## Core Rule

Claims must stay smaller than proof.

## Evidence Tiers

### Tier 0 - Point Of View
- what AGC believes
- what patterns AGC sees
- what problems AGC is oriented to solve

### Tier 1 - Operator Proof
- internal operating systems
- process rigor
- prototypes, playbooks, templates, or demonstrations

### Tier 2 - Adjacent Proof
- directly relevant experience, partner context, or operator evidence that informs the offer
- must be framed carefully and not presented as AGC-delivered contract history

### Tier 3 - Direct AGC Proof
- real AGC-delivered work, outcomes, or references
- only use when the proof actually exists

## Claim Formula

Every capability line should be expressible as:

`AGC helps <buyer type> with <problem shape> using <approach>, with proof anchored in <evidence tier>.`

If the last clause cannot be completed honestly, the line is not ready.

## Minimum Capability Surface

Any first-pass capability note should include:

- buyer type
- problem shape
- delivery motion
- proof line with tier or qualifier
- explicit limitation when proof is still adjacent
- bounded next step

## Approved Language Patterns

- "AGC is building around..."
- "AGC can support..."
- "AGC is positioned to help with..."
- "Current proof is strongest in..."

## Prohibited Language Patterns

- "AGC has delivered this for federal agencies" when it has not
- "AGC is certified/cleared/compliant" without actual proof
- "AGC guarantees..." for outcomes it cannot control
- any statement that implies past performance AGC cannot evidence

## What To Build Next

The first real capability surface should probably be:

- one-page positioning note
- short FAQ for buyer objections
- email-ready response snippet library

Each should cite the exact proof boundary instead of smoothing it over.
